Блок 1: Проектирование API отельного продукта

Минимальный набор методов, необходимых для обеспечения основных бизнес-процессов системы бронирования отелей:

Функциональность HTTP-метод Эндпоинт Обоснование
Поиск отелей GET /api/hotels Базовый поиск с фильтрами (город, даты, гости). Без поиска невозможно начать процесс Бронирования.
Детали отеля и его доступность GET /api/hotels/{id}/rooms Получение информации о конкретном отеле, типах номеров, ценах и наличии на выбранные даты.
Создание сессии бронирования POST /api/booking-forms Резервирование номера на короткое время, получение book_hash для продолжения. Защищает от двойного бронирования.
Подтверждение бронирования POST /api/orders Финальное создание заказа с данными Гостей и оплаты. Это ключевой бизнес-процесс.
Получение информации о брони GET /api/orders/{id} Просмотр деталей заказа, статуса, условий отмены. Необходимо для информирования Гостя.
Отмена бронирования DELETE /api/orders/{id} Важная функция для управления заказами, реализована через DELETE как идемпотентный метод.

Обоснование данного набора методов и эндпоинтов

Этот набор методов покрывает полный пользовательский путь (Customer Journey) от поиска отеля до отмены брони. Он включает двухэтапное бронирование (Сессия + Подтверждение), что является отраслевым стандартом для безопасности финансовых транзакций.

Блок 3: Чтение логов и ответ Партнёру

Кейс 1: Ошибка "sandbox_restriction" при вызове Booking Form

Технический анализ

Данные запроса: "book_hash": "h-c8833e24-1d3c-56ba-85a4-ca873e31f6fe"

Ответ API: "error": "sandbox_restriction"

Источник: Документация Sandbox, раздел "Create booking process"

Причина: Конфликт окружений. Запрос с использованием API-ключа Sandbox (Песочницы) отправлен для отеля, который не является тестовым, либо book_hash из Production (Продакшена) используется в Песочнице (или наоборот).

Ответ Партнёру (пример)

Тема: Разбор ошибки «sandbox_restriction» при вызове Booking Form

Здравствуйте!

Благодарим за обращение. Мы проанализировали ваш запрос и определили причину ошибки.

Краткое описание: Конфликт окружений Sandbox/Production.

Эта ошибка возникает, когда запрос с использованием Sandbox API-ключа отправлен для отеля, который не является тестовым, либо когда хэш бронирования (book_hash) из Production используется в Sandbox (или наоборот).

Рекомендуемые действия для решения:

  1. Убедитесь, что все вызовы API в рамках одного потока бронирования идут на один базовый URL: либо на https://sandbox.emergingtravel.com (для тестов), либо на https://api.emergingtravel.com (для рабочих данных).
  2. Проверьте, что ID отеля, с которым работаете, есть в списке тестовых отелей Sandbox. Бронировать в Sandbox можно только их.

Если проблема сохранится, пришлите, пожалуйста, полный лог запроса, включая заголовки.

Кейс 2: Ошибка "booking_form_expired" при вызове Order Booking Finish

Технический анализ

Данные запроса: "partner_order_id": "100211111111111"

Ответ API: "error": "booking_form_expired"

Источник: Документация Sandbox, раздел "Start booking process"

Причина: Поле partner_order_id содержит или имитирует специальный триггер _booking_form_expired, что заставляет систему возвращать эту ошибку. Также возможно истинное истечение времени сессии (15-20 минут между Booking Form и Booking Finish).

Ответ партнёру (образец)

Тема: Разбор ошибки «booking_form_expired» при вызове Order Booking Finish

Здравствуйте!

Проанализировали ваш запрос на завершение бронирования. Ошибка «booking_form_expired» означает, что сессия бронирования истекла.

Детали: В вашем запросе обнаружено поле "partner_order_id": "100211111111111". Если этот ID был сформирован с использованием суффикса _booking_form_expired, наша система специально имитирует ситуацию истёкшей сессии — это стандартный способ тестирования такого сценария. Альтернативная причина — превышение времени между вызовами методов Booking Form и Order Booking Finish (стандартный таймаут 15-20 минут).

Рекомендуемые действия для решения:

  1. Проверьте логику формирования partner_order_id в вашей системе. Убедитесь, что для реальных броней не используется тестовый суффикс.
  2. Убедитесь, что между вызовом метода Booking Form и Order Booking Finish проходит не более 15-20 минут.
  3. Реализуйте retry-логику с получением нового book_hash в случае истечения таймаута.

Если вы не использовали тестовый суффикс, пришлите, пожалуйста, точное время обоих запросов — мы проверим логи на нашей стороне.

С уважением,
Команда поддержки API.

Блок 4: Анализ использования API

Ситуация: Партнёр запустил промо-акцию на отели в Краснодарском крае, и случилась пиковая нагрузка 487 запросов в минуту.

Вопрос Ответ Обоснование
1. Какой метод использовался? /api/b2b/v3/search/serp/region/ Промо-акция на отели в определённом регионе (Краснодарский край) предполагает поиск по региону, а не по конкретным отелям.
2. Будет ли превышен лимит? Да, превышен Лимит: 300 RPM, пиковая нагрузка: 487 RPM
487 / 300 = 162.3% (превышение на 62.3%)
3. Какую ошибку вернёт метод? HTTP 429 Too Many Requests Стандартный HTTP-статус для исчерпания лимита запросов. Может включать заголовок Retry-After.

Рекомендации партнёру

Краткосрочные меры:

  • Реализовать механизм повторных попыток с задержкой для запросов, получивших ответ 429.
  • Добавить кэширование результатов поиска по популярным регионам на 1-2 минуты на своей стороне.

Долгосрочные меры:

  • Обсудить с аккаунт-менеджером возможность увеличения лимита для данного метода. Это в случае, если акции/распродажи проходят регулярно, где случается превышение лимитов.
  • Реализовать предварительную загрузку данных для промо-страниц.

Рекомендации внутренней команде

Для команды интеграций и аккаунт-менеджеров:

  • Настроить мониторинг и систему оповещений при достижении 80% от лимита (240 RPM для метода /region/).
  • Рассмотреть внедрение гибких лимитов, позволяющих кратковременные пики выше среднего.
  • Для Партнёров, которые планируют промо-акции, предусмотреть процедуру временного увеличения лимитов.
