## Кейс
_"Представь, что активный партнёр недоволен возможностями API и просит доработать его. Опиши свои действия в такой ситуации."_

## Мой процесс коммуникации с Партнёром (по ITIL):

1) Эмпатия и сбор информации: Первый ответ, запрос логов, ID запросов.

2) Анализ, классификация проблемы: Является ли это багом, доработкой или непониманием документации?

3) Внутренняя коммуникация: Создание тикета (в Jira) для разработки, общение с продакт-менеджером.

4) Коммуникация с Партнёром: Регулярные апдейты, реалистичные сроки, предложение workaround (если есть).

5) Завершение: Информирование об устранении, обновление документации, выводы для предотвращения.
